---
title: Image State API
---

## Overview

The editor persists the project's canvas transform (one row per
project) using **µpx fixed-point integers** so the canvas can be
restored without drift.

State is anchored at the project's **master image id** (PR #124).
Every write resolves the persistence target to `master.id`; the
`image_id` the client sends is treated as informational (used for
the in-project + lock-guard check on the surface the user was
operating on) and is **not** the storage key.

Save triggers (important):

- Saves happen **only** on explicit user actions (drag end /
  transform end / input commit).
- Unit dropdown changes are display-only and must not trigger any
  save.
- Rounding happens **exactly once** (bake-in / user input
  conversion). Save/persist must not apply a second rounding pass.

MVP guarantees:

- No drift/flash in displayed sizes after reload.
- State survives every filter-base-copy recreation, chain reset,
  or trace tombstone (because it lives at master.id, not at the
  editor surface's id).
- Unit changes never trigger persistence.
- Persisted state is always µpx canonical values.

Endpoint:

- `GET /api/projects/:projectId/image-state`
- `POST /api/projects/:projectId/image-state`

Auth:

- Requires an authenticated Supabase user (cookie-based auth via
  SSR client).
- Project access is enforced via RLS and an explicit
  "project accessible" check in the route handler.

## Data model (DB)

Table: `public.project_image_state`

Key: `(project_id, image_id)` where `image_id` is the project's
`master.id`. The single row per project semantic is a route
invariant, not a DB constraint — direct DB writes that resolve to
a different image_id bypass it.

Fields:

- **canonical µpx size**: `width_px_u`, `height_px_u`
  (string BigInt, required)
- **optional µpx position**: `x_px_u`, `y_px_u` (string BigInt)
- **rotation**: `rotation_deg` (number)

## GET

- Server calls `getProjectMasterImageRow(projectId)` and looks up
  state at `(project_id, master.id)`.
- No query parameters (legacy `?imageId=` was dropped in the
  master-anchor client cleanup).

Response (state exists):

```json
{
  "exists": true,
  "state": {
    "image_id": "uuid-of-master",
    "x_px_u": "123450000",
    "y_px_u": "67890000",
    "width_px_u": "1024500000",
    "height_px_u": "768250000",
    "rotation_deg": 0
  }
}
```

Response (no state row):

```json
{ "exists": false, "state": null }
```

## POST

Request body:

```json
{
  "x_px_u": "123450000",
  "y_px_u": "67890000",
  "width_px_u": "1024500000",
  "height_px_u": "768250000",
  "rotation_deg": 0
}
```

Legacy fields `image_id` and `role` are accepted by the validator
but **ignored**. Older client builds still in flight during a
deploy window will include them; the server resolves the
persistence key and the lock-guard target from `projectId` alone.
A follow-up release should drop the legacy fields from the
validator once those builds are gone.

Validation steps (in order):

1. Transform fields (`width_px_u`, `height_px_u`, `rotation_deg`)
   must be present and within bounds.
2. `getProjectMasterImageRow(projectId)` must return a master row.
   Returns `409 no_master_image` otherwise.
3. That master row must not be locked. Returns `409 lock_conflict`
   if it is.
4. Per-axis preservation: when `x_px_u` or `y_px_u` is omitted,
   the route reads the current row at master.id and fills in the
   unchanged axis so a position-only edit doesn't clobber the
   other axis.
5. Upsert at `(project_id, master.id)`.

Notes:

- `width_px_u/height_px_u` are required on every write.
- No runtime fallback/migration is supported. If
  `width_px_u/height_px_u` come back missing, the GET returns
  `400 schema_missing`.
- Bounds: sizes are clamped/validated to `1px..32768px` (in µpx:
  `1_000_000..32_768_000_000`).
